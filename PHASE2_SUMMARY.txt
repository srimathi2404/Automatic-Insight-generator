================================================================================
USAGE EXAMPLE (in development)
================================================================================

from query_generation_agent import QueryGenerationAgent
from insight_generator import InsightGenerator
from dashboard_generator import DashboardGenerator
from models import InvestigationType

# Initialize
qga = QueryGenerationAgent(trino_client, data_explorer)
ig = InsightGenerator()
dg = DashboardGenerator()

# Investigation
results = qga.investigate_objective(
    table="events",
    objective="Find top revenue sources",
    investigation_type=InvestigationType.TOP_CONTRIBUTORS,
    dimensions=["source"],
    metrics=["revenue"]
)

# Generate insights
insights = ig.generate_insights(
    findings=results["findings"],
    investigation_type=InvestigationType.TOP_CONTRIBUTORS,
    objective=results["objective"],
    result_data=results["results"][0]
)

# Generate dashboard
dashboard = dg.generate(
    insights=insights,
    query_results=results["results"],
    executive_summary=ig.generate_executive_summary(insights)
)

# Export
dg.export_html(dashboard, "report.html")
dg.export_json(dashboard, "report.json")

================================================================================
DESIGN DECISIONS
================================================================================

1. Simple over Complex
   - Minimal HTML generation (no external chart libraries)
   - Direct table display from DataFrames
   - Focus on data availability over visualization

2. Evidence-Based Insights
   - All insights include observation, evidence, recommendation
   - Confidence scores guide decision-making
   - No speculation or AI-generated content (Phase 3 task)

3. Type-Specific Generation
   - Different insight types generate appropriate messages
   - Trend analysis shows % change
   - Top-N analysis uses Pareto principle
   - Distribution analysis shows range

4. Modular Design
   - Each component can be used independently
   - All classes follow single responsibility
   - No tight coupling between modules
